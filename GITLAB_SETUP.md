# GitLab Pages Setup Guide

This guide ensures your Synthwave Notes app deploys correctly to GitLab Pages.

## 📋 Prerequisites

- GitLab account
- Git installed locally
- Node.js 18+ (for local builds only)

## 🚀 Quick Setup

### 1. Create GitLab Repository

```bash
# Clone your new repository
git clone https://gitlab.com/YOUR_USERNAME/synthwave-notes.git
cd synthwave-notes

# Copy all files from this project
# Add all files
git add .
git commit -m "Initial commit"
git push origin main
```

### 2. Enable GitLab Pages

1. Go to your project in GitLab
2. Navigate to **Settings → Pages**
3. Pages should automatically be enabled when pipeline runs

### 3. First Deployment

After pushing, GitLab CI/CD will:
1. Install dependencies
2. Build your site
3. Deploy to GitLab Pages

Your site will be available at:
```
https://YOUR_USERNAME.gitlab.io/synthwave-notes/
```

## 📝 Understanding expire_in

The `expire_in` setting in `.gitlab-ci.yml` controls how long GitLab keeps artifacts:

### Build Job Artifacts
```yaml
artifacts:
  paths:
    - public
  expire_in: 1 hour  # ← Temporary artifact
```
- **Purpose**: Passes built files to the pages job
- **Duration**: 1 hour is sufficient
- **Why short**: These are temporary files only needed between jobs

### Pages Job Artifacts
```yaml
pages:
  artifacts:
    paths:
      - public
    expire_in: never  # ← Your deployed website
```
- **Purpose**: This IS your deployed website
- **Duration**: Set to `never` for production
- **Options**:
  - `never` - Site stays up forever (recommended)
  - `30 days` - Site expires after 30 days
  - `1 year` - Site expires after 1 year
  - `1 week` - For temporary demos

⚠️ **IMPORTANT**: If pages artifacts expire, your site goes offline!

## 🔧 GitLab-Specific Configuration

### Required Structure
```
project/
├── .gitlab-ci.yml    # CI/CD pipeline (required)
├── public/           # Created by CI/CD (don't commit)
│   └── index.html   # Deployed file
└── ... other files
```

### Key Requirements

1. **Job Name**: The deployment job MUST be named `pages`
2. **Directory**: Files MUST be in `public/` directory
3. **Artifacts**: The pages job MUST specify artifacts

### Pipeline Stages

1. **Build Stage**: 
   - Installs dependencies
   - Builds the site
   - Creates public/ directory
   - Copies index.html to public/

2. **Deploy Stage**:
   - Takes artifacts from build
   - Deploys to GitLab Pages
   - Makes site available at your URL

## 🛡️ Security & Access

### Private Projects
If your repository is private:
1. Go to **Settings → General → Visibility**
2. Under "Pages", you can set access level:
   - **Everyone**: Public access (default)
   - **Only Project Members**: Requires GitLab login

### Custom Domains (Optional)
1. Go to **Settings → Pages**
2. Click "New Domain"
3. Add your domain and follow DNS instructions

## 🐛 Troubleshooting

### Pipeline Failed
Check the pipeline logs:
1. Go to **CI/CD → Pipelines**
2. Click on the failed pipeline
3. Click on the failed job
4. Read the error messages

Common issues:
- **npm ci failed**: Delete `package-lock.json` and run `npm install` locally
- **public directory not found**: Ensure build creates the directory
- **pages job not found**: Job must be named exactly `pages`

### Site Not Accessible
1. Check if pipeline succeeded
2. Go to **Settings → Pages** to see the URL
3. Wait 10-30 minutes for first deployment
4. Check artifact expiration (shouldn't be expired)

### 404 Errors
The 404.html file is created as a copy of index.html, so all routes work correctly.

## 📊 Pipeline Optimization

### Caching
The pipeline caches `node_modules/` based on `package-lock.json`:
- Faster builds
- Less bandwidth usage
- Consistent dependencies

### Build Times
Typical pipeline duration:
- First run: 2-3 minutes (downloads dependencies)
- Subsequent runs: 30-60 seconds (uses cache)

## 🔄 Updating Your Site

1. Edit your markdown files locally
2. Run `npm run build` to test (optional)
3. Commit and push:
   ```bash
   git add .
   git commit -m "Update notes"
   git push
   ```
4. GitLab automatically rebuilds and deploys

## 💡 Tips

1. **Branch Strategy**: 
   - Use `main` for production
   - Use merge requests for testing changes

2. **Local Testing**:
   ```bash
   npm run build
   npm run serve
   # Visit http://localhost:8080
   ```

3. **Monitor Usage**:
   - Check **Settings → Usage Quotas**
   - GitLab provides 10GB storage for Pages

4. **Backup Strategy**:
   - Your source files are in Git
   - Built site can be regenerated anytime
   - No need to backup public/ directory

## 🚫 What NOT to Do

1. **Don't commit public/ directory** - It's generated by CI/CD
2. **Don't change the pages job name** - Must be exactly "pages"
3. **Don't set pages artifacts to expire** - Unless you want site to go offline
4. **Don't remove .gitlab-ci.yml** - Required for GitLab Pages
